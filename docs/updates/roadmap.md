# 覇道ライブラリ 3.0.0.0 全体ロードマップ

## 目的
型検索を、単なる検索機能ではなく、部隊編成画面で完成形を検討するための候補提示機能へ段階的に拡張する。

## 基本方針
- 型検索は候補表示を担当し、組み合わせ検討と最終登録は部隊編成画面で行う。
- 主将、副将、補佐、侍従、装備、陣形、兵器、名馬、軍馬技能の9役割を対象とする。
- 既存の部隊編成成立ゲートを迂回しない。直接配置は行わず、既存選択処理へ委譲する。
- 既存保存データを上書きしない。型検索から作成する部隊は新規部隊として追加する。
- 全データ表示と保存データ表示を分離し、保存データ表示では所有情報と設定値を反映して候補を絞り込む。
- HTMLを肥大化させない。新規責務は外部JSへ分離し、HTMLは読込宣言と最小限の初期化に留める。
- JSファイル名はUpdate番号ではなく責務名を使用する。`updateNN_xxx.js`、`xxx_updateNN.js`、`temp_xxx.js`、`new_xxx.js` のような一時命名は禁止する。
- 既存責務へ統合できる場合は、責務一致する既存JSへ追記する。新規JS作成時は既存ファイルへ統合できない理由を記録する。
- アプリ開発ブランチへのPush後はプレビューへ自動同期する。

### Update06: 残り4役割の候補トレイ接続
**状態:** 完了

- 陣形、兵器、名馬、軍馬技能を候補トレイから既存選択処理へ接続した。
- 9役割すべてで、候補一覧 → 候補トレイ → 既存選択処理の経路を統一した。
- 陣形は既存陣形マスタ選択と侍従3×3修復処理へ委譲する。
- 兵器は既存兵器マスタ選択とLv正規化処理へ委譲する。
- 名馬は所有済み軍馬を既存の軍馬3枠割当処理へ委譲する。
- 軍馬技能は対象技能を持つ所有済み軍馬を探索し、既存の軍馬3枠割当処理へ委譲する。
- 主将、副将、補佐、侍従、装備では既存選択ダイアログを維持し、兵科、射程、侍従3×3条件、装備重複の成立ゲートを迂回しない。
- 実装は責務名ファイル `hado_candidate_tray.js` と `hado_candidate_tray_core.js` に統合し、Update番号付きJSファイルは残さない。

### Update07: 適合スコア表示と評価ロジック調整
**状態:** 完了

- 型ごとの採点項目は5項目、重みなしとする。
- 技能Lvは採点しない。
- 兵力は採点対象に含める。
- 型充足度という別指標は設けない。
- 適合スコアは10段階で理解しやすく表示する。
- 殲滅系のスコアが全体的に高くなりすぎないよう補正する。
- 主軸理由と補助理由を表示する。
- Update08で所有情報フィルター適用後にも同一ロジックを再利用できるようにする。

### Update08: 保存データ対応型編成ナビ、新規部隊作成、保存評価、履歴、グループ管理
**状態:** 完了（最新: Update08.22）

- 保存データ表示モードで型編成ナビを利用できるようにする。
- 全データ表示では全武将、全装備、全名馬などを候補対象とする。
- 保存データ表示では、所有登録済みの武将、装備、名馬などに候補を限定する。
- 保存データに登録された将星、技能Lv、装備所有状況などの設定値を候補表示と評価へ反映する。
- 型検索から新規部隊を作成する際、選択中の型を自動入力する。
- 既存部隊を上書きせず、新規部隊として追加する。
- 編成結果に評価型ID、トータルスコア、評価スコア、マイメモを保持する。
- 1部隊あたり最大10件の履歴を保持する。
- 1グループあたり最大12部隊、最大5グループを管理する。

## Update記録ルール
- 各Updateは `docs/updates/updateNN/roadmap.md`、`implementation.md`、`report.md` に記録する。
- 枝番修正は `Update05.1` のように管理する。
- 実装時は本ロードマップとの差分を `implementation.md` に記載する。
- 完了時は確認済み、未確認、未解決、次Update候補、プレビュー反映結果を `report.md` に記載する。
- 新規JS作成時は責務名を使用し、Update番号、一時作業名、枝番をファイル名に含めない。
